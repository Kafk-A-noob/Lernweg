# Next.js 学習ログナレッジサイト 構築プラン検討書

本資料は、現在ローカルに存在するMarkdownドキュメント群（Learning_Log）を、Next.jsを用いてセキュアなWebナレッジサイトとして外部公開するためのアプローチごとのメリット・デメリット、および技術スタックの選定に向けた検討事項をまとめたものである。

---

## 1. 全体要件確認

- **対象**: `Learning_Log` 内のMarkdownファイル（ディレクトリによる階層構造あり）
- **フレームワーク**: React / Next.js (App Routerベース)
- **公開レベル**: 「URLを知っている」かつ「認証を通った」特定の人間（採用担当者や関係者）のみが閲覧可能とする限定公開。

---

## 2. サイト構築手法（フロントエンド・アーキテクチャ）の比較

MarkdownファイルをWebページ化するにあたって、実務現場でも使われる「2つの主流な手法」を比較する。

### [プランA] 圧倒的特化型：『Nextra』を利用する手法

Vercel公式が提供するドキュメント作成特化のNext.jsフレームワーク「Nextra」を導入する。

- **概要**: Next.jsの中に `nextra` パッケージを入れ、指定の形式でマークダウンを配置するだけで、美しいドキュメントサイトが自動生成される。
- **メリット**:
  - **構築速度**: ゼロから作る手間の1/10で、サイドバー（目次）、全文検索機能、ダークモードトグル、シンタックスハイライト（コードの色付け）といった、ドキュメントに必須な機能がすべて最初から備わっている。
  - **更新性**: 今後、新しい `.md` ファイルをポイッと追加するだけで、自動で目次に反映されページが生成される。運用が極めて楽。
- **デメリット**:
  - **ブラックボックス化**: 「なぜマークダウンが画面に描画されるのか」という内部のコア処理をNextraが隠蔽してしまうため、Next.js自体の深い学習効果（Server Componentsでの読み込みや動的ルーティングの知識）は薄まる。
- **評価**: 「早く、高品質で実用的なナレッジサイトを運用したい」という目的におけるベストプラクティス。

### [プランB] 完全学習特化型：『App Router + 自作パース』手法

ライブラリに頼りきらず、Next.jsの最新機能（App Router）を使って自作する。

- **概要**: `fs`（Node.jsのファイル読み込み）モジュールでローカルの `.md` ファイルを読み取り、`react-markdown` 等のライブラリを使って自分自身でHTMLに変換して画面に描画する。
- **メリット**:
  - **技術の完全理解**: 「動的ルーティング（`[slug]`）の仕組み」「Server Componentsでの非同期データ取得（`fs.readFile`）」「静的サイト出力（SSG）」といった、Next.jsの核心とも言える強力なエンジンを、文字通り自分の手で組み上げる経験が積める。
  - **完全な自由度**: 画面のデザインや構成を1ミリ単位でTailwind CSSなどを使って自作のポートフォリオと統一できる。
- **デメリット**:
  - **工数の肥大化**: 「目次の自動生成」「検索機能」「リンクの制御」といった付属機能まで全部自作する必要があり、完成までに多大な労力がかかる。
- **評価**: 「これを作る過程そのものをNext.jsの学習・ポートフォリオのアピール材料にしたい」という明確な意思がある場合の最高の教材。

## 3. アクセス制限手法（セキュリティ制御）の比較

「関係者しか見れない」状態を作るためのアプローチ。

### [手法1] Next.js Middleware による Basic認証 (推奨)

- **概要**: プロジェクトの直下に `middleware.ts` を1ファイル置くことで、サイト全体へのアクセスを検知し、ブラウザ標準の「ID・パスワード入力画面」を強制表示させる。
- **メリット**:
  - データベースが不要。パスワードは環境変数（`.env` 等）にセットするだけで済む。
  - 「URLは教えるが、パスワードは面接や提出時にお伝えする」という運用が非常にスマートに行えるため、実務の開発デモ環境（ステージング環境）で最も多用される。
- **デメリット**:
  - ユーザーごとの細かい権限管理（Aさんは見れるがBさんは見れない等）はできない（全員共通のID/Passとなる）。

### [手法2] Server Actions + Cookie による自前ログイン画面の実装

- **概要**: 本格的な「ログイン画面（`app/login/page.tsx`）」を作り、パスワードが合致すればCookieを付与して本編へのアクセスを許可する。
- **メリット**:
  - ポートフォリオとして「認証機能（Session / Cookie管理）の実装ができる」というアピールに繋がる。
  - ログイン画面のデザインを自由に作れる。
- **デメリット**:
  - 実装の難易度が大きく跳ね上がる。MiddlewareのBasic認証であれば10分で終わるものが、数時間〜数日の開発タスクとなる。

## 4. 総括と次のアクション提案

**【提案する構成パス】**

もし、「Next.jsのコア機能を深く理解するための学習」に振り切る（＝このサイトの制作過程そのものをポートフォリオの一部にする）のであれば、以下の構成を推奨します。

1. **ベース**: `Next.js (App Router) + Tailwind CSS` （プランBの自作アプローチ）
2. **描画機構**: `react-markdown` 等を用いたマークダウンパース機構の自作
3. **セキュリティ**: `Middleware` による Basic認証の導入（手法1）
4. **デプロイ**: `Vercel` への無料自動デプロイ

**【ユーザー様への確認事項】**

1. 現状のゴールとして、**「サクッと見栄えの良い運用サイトが欲しいか（プランA）」**、それとも**「学習とポートフォリオの実績作りのために泥臭く自作したいか（プランB）」**、どちらのモチベーションが強いでしょうか？
2. 今後このプロジェクトを作っていくにあたり、初期設定（`npx create-next-app` 等のコマンド発行）からサポートを開始してもよろしいでしょうか？
