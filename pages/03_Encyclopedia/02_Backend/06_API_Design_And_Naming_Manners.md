# REST API設計と命名規則の作法

## 1. 命名規則（Naming Conventions）とリーダブルコード

- **【原則】**：コードは「書く」時間より「読まれる」時間の方が圧倒的に長いです。「誰が見ても一瞬で目的がわかる」命名が求められます。
- **作法1: 役割ベースの命名**：変数名は `data` や `temp` のような無意味な名前を避け、`userList` や `isLoggedIn` のように中身と型が推測できる名前にします。
- **作法2: コメントは「何（What）」ではなく「なぜ（Why）」**：コードを読めば分かる処理内容を書くのではなく、「なぜこの複雑なビジネスロジックにしたのか」の理由をコメントに残します。

## 2. REST API設計のお作法

- **【原則】**：URLは「リソース（名詞）」を表し、操作は「HTTPメソッド（動詞）」で表すのがRESTの基本です。
- **NG例**: `GET /createUser` や `POST /updateUser?id=1`
- **OK例**: `POST /users` (作成) / `PUT /users/1` (更新) / `DELETE /users/1` (削除)
- **ステータスコードの使い分け**: 正常時は `200 OK` (作成時は `201 Created`)。クライアント側のミス（バリデーションエラー等）は `400 Bad Request`、サーバー側のダウンは `500 Internal Server Error` と、エラーの責任の所在を明確に分離します。
