# Git運用・CI/CDの作法

## 1. Gitのコミットとブランチ運用

- **ブランチ戦略 (GitHub Flow等)**
  メインブランチ（`main`）は「常に正常に動く、本番にデプロイ可能な状態」を維持します。作業する際は必ず `feature/xxx` などのブランチを切り、完成したらPull Request（PR）を出してレビューを受け、問題なければマージします。
- **コミットメッセージの作法**
  `fix` や `update` だけのメッセージは後から「何を直したか」が全く追えません。
  プレフィックスをつけて構造化するのがモダンなマナーです。
  - `feat`: 新機能の追加
  - `fix`: バグ修正
  - `refactor`: リファクタリング（動作を変えずに中身整理）
  - `docs`: ドキュメント（README等）の修正

## 2. CI/CD (継続的インテグレーション/デリバリー) のマナー

- **CI（テスト自動化）の義務**
  PRが出された時、GitHub Actions等を通じて自動で「Linter（文法チェック）」や「Unitテスト」を走らせます。これが通らないPRは、レビュアー（人間）が見る価値がないため、自動で弾く（Merge不可にする）のが鉄則です。
- **CD（自動デプロイ）**
  `main` ブランチにマージされた瞬間、自動で本番環境やステージング環境へコードが反映される仕組み（GitOpsなど）を作ります。人間が手作業でサーバーにファイルを上げる「FTPアップロード」は人的ミスの温床となるため駆逐すべき対象です。
