# 多段SSHトンネルによるセキュアなインフラ接続設計

本稿では、手元の秘密鍵をインターネットに公開されている踏み台サーバーに置くことなく、セキュアに「奥の奥」にあるプライベートサブネット内のRDS等へGUIから接続するための、実務的かつ強固なネットワーク設計思想について解説します。

## 1. 背後にある設計思想（Why）

なぜ、面倒な「多段トンネリング（ポートフォワーディング）」を行ってまで接続経路を構築するのか。その背後には、現代のインフラ構築に欠かせない以下の重要なセキュリティパラダイムが存在します。

### ① 秘密鍵の局所化（Zero Trustの第一歩）
踏み台サーバー（Bastion Host）は、不特定多数（他の開発メンバーや外部の攻撃者など）の目に触れ、常に攻撃の脅威に晒される可能性があります。
もし踏み台サーバー上に秘密鍵を配置してしまうと、踏み台が突破された瞬間に内部システム全体へのアクセス権（鍵）を奪われることになります。
**「秘密鍵は絶対にローカルPCから出さず、ローカルからの署名のみで認証を完結させる」**こと。これが鍵の流出リスクを物理的にゼロにするための鉄則です。

### ② ポート番号の隠蔽
SSH標準の「22番ポート」は、世界中の攻撃者から無差別なブルートフォース攻撃（総当たり攻撃）を絶えず受けています。
これを「2222番」など、デフォルトから外したカスタムポートに変更するだけで、Botによるノイズ的な攻撃の大半を回避し、ログの治安を保つことができます。

### ③ 多段フォワーディングの活用（透過的なアクセス）
1つ目のトンネル（L1トンネル）で「インフラの土台」を作り、2つ目のトンネル（L2トンネル）で「アプリケーションの通信」を通す。この多段構成にすることで、複雑なAWSネットワーク（VPCやサブネット）の構造を意識することなく、あたかもローカルにDBが存在するかのような操作感で開発を進めることが可能になります。
