# チーム開発の作法とセキュリティ常識

## 1. Pull Request（レビュー）の出し方・受け方案

- **【出す側（作成者）のマナー】**
  - **巨大なPRを作らない**：1つのPRで「新機能追加＋リファクタリング＋関係ないバグ修正」を混ぜると、レビュアーの脳への負担が限界を超えます。数日でマージできる小さな単位（1チケット＝1PR）に分割します。
  - **レビュアーへの道しるべ**：PRの概要欄に「何を解決したか」「テストはどうやったか」「特に見てほしい箇所」を明記します。
- **【受ける側（レビュアー）のマナー】**
  - **人格攻撃をしない**：「なんでこんなコード書いたの？」ではなく、「この処理はXの時にバグになる可能性があるため、Yに変更するのはどうでしょうか」と、リスペクトを持って提案します（心理的安全性）。

## 2. Webセキュリティの「最低限の常識」

- **SQLインジェクション**
  ユーザーの入力値（フォーム等）を、そのまま生のSQLに文字列結合してはいけません。必ず「プレースホルダ（バインド変数）」や、安全なORMを使用します。
- **XSS (クロスサイトスクリプティング)**
  ユーザーが入力した文字を画面に表示する際、`<script>` などのタグが暴発しないように、必ずHTMLエスケープ（サニタイズ）処理をかけます（ReactなどのモダンFWは基本自動でやってくれます）。
- **CORSエラー（フロントエンドの壁）**
  異なるドメイン間でAPI通信を行う際に出るセキュリティエラーです。バックエンド側で「許可するドメイン」を明示的に設定（CORSヘッダーの設定）して解決するのがお作法です。
