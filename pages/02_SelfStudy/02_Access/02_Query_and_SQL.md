# 02. クエリ設計とSQL的視点

Accessにおける「クエリ」は、バックエンドにおけるAPI Routesや、PrismaなどのORMでデータ検索を行う処理そのものです。これをGUI操作で作れるのがAccessの強みですが、背後ではしっかり **SQL** が動いています。

## 1. クエリの種類と役割

### 選択クエリ (Select Query)

もっとも基本となるクエリです。画面上にデータを呼び出したり、帳票（レポート）に印字する元のデータセットとして使われます。
SQLの `SELECT * FROM TableName WHERE Condition;` と完全に同一です。

### アクションクエリ (Action Query)

データベースのデータを直接書き換える、副作用のあるクエリ群です。Web APIの `POST` / `PUT` / `DELETE` に相当します。

- **追加クエリ**: 他のテーブルのデータを抽出して、別のテーブルに `INSERT` します。
- **更新クエリ**: 条件に一致するレコードの値を一括で `UPDATE` します。（バッチ処理等で使います）
- **削除クエリ**: 条件に一致するレコードを `DELETE` します。（※物理削除になります。Accessで論理削除を実装するには、テーブルに `isDeleted` フラグカラムを持たせ、更新クエリを叩く設計にします）

## 2. デザインビューとSQLビュー

Accessのクエリエディタには 2つのビュー（視点）があります。

1. **デザインビュー**: テーブルをドラッグ＆ドロップし、GUIで結合（JOIN）や抽出条件を組み立てる画面です。文法エラーを自動で防いでくれます。
2. **SQLビュー**: 実際に実行されるSQL文をテキストベースで確認・編集できる画面です。

**エンジニア向けのヒント:**
複雑なクエリや、他のシステム（MySQLやPostgreSQLなど）へ移植可能なSQLを書く場合は、デザインビューで大枠を作ってから、SQLビューに切り替えて微調整を行うというアプローチが有効です。

## 3. クエリの抽出条件 (Where句相当)

デザインビューの「抽出条件」パラメータは、そのままSQLの `WHERE` 句になります。

- **Like演算子とワイルドカード**:
  `Like "*山田*"` と入力すると、名前が「山田」を含むあいまい検索になります。
  （注意: Access独特の仕様として、標準SQLの `%` ではなく `*` がワイルドカードとして使われます）

- **パラメータクエリ**:
  抽出条件に `[都道府県を入力してください]` のようにブラケット(`[]`)で囲んだテキストを記述すると、実行時に都度ユーザーへ入力を促す小さなダイアログ（プロンプト）が出現します。
  実践的にはダイアログよりも、「フォーム上のテキストボックスの値を参照する」 `[Forms]![フォーム名]![テキストボックス名]` という設計が最も多用されます。
