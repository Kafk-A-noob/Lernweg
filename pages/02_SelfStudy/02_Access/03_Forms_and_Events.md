# 03. フォームUI操作とイベントプロシージャ

Accessの「フォーム」は、ユーザーがデータを入力したり閲覧するための画面（フロントエンド・UI）です。Reactにおける「コンポーネント」や「ページ」と類似の役割を果たします。

## 1. フォームのバウンドとアンバウンド

Accessのフォームを理解する上で最も重要な概念が**「バウンド（連結）」**です。

### バウンドフォーム（連結フォーム）

フォーム自身が特定のテーブルやクエリとダイレクトに裏で繋がっている状態です。
- 画面上のテキストボックスのプロパティである「コントロールソース」にDBのカラム名（例：`UserName`）を紐づけます。
- デメリットとしては、ユーザーがテキストボックスの文字を打ち替えた瞬間に**DBの値が自動的に同期（上書き）されてしまう**という強力すぎる副作用を持っています。
- Web開発での「2Wayデータバインディング」に近く、保存ボタンを作らなくても状態が永続化されます。

### アンバウンドフォーム（非連結フォーム）

フォームがどのテーブルとも繋がっていない状態です。
- いったん画面上の仮のテキストボックスに入力させ、ユーザーが「保存」ボタンを押した際（イベントハンドラ内）でSQLを発行し、自前でDBに `INSERT` や `UPDATE` を行う設計です。
- ReactでフォームのStateを管理し、onSubmitでAPIへPOSTする モダンなWebパラダイムと一致しており、複雑な業務アプリではこちらを採用することが多いです。

## 2. イベントプロシージャ (Event-Driven)

画面上で「ユーザーが何かのアクションを起こした瞬間」をフックして動作するVBAの処理を「イベントプロシージャ」と呼びます。（JSの `addEventListener` の概念です）

例：ボタンをクリックした時のイベント (`onClick` 相当)

```vb
Private Sub btnSave_Click()
    ' 保存処理などのロジック
    MsgBox "データを保存しました"
End Sub
```

### 代表的なイベントの種類

1. **Click (クリック時)**: もっとも多用される、ボタンを押した時のイベント。
2. **BeforeUpdate (更新前処理)**: ユーザーがテキストボックスの文字を書き替え、別の場所をクリックした瞬間に走るイベント。保存する前に「入力文字数の制限」や「未入力のバリデーション（Validation）」を行うために使います。ここで `Cancel = True` を返すと、入力を無効化できます。
3. **Form_Load (読み込み時)**: フォームが開かれた直後（Reactの `useEffect` マウント時）に走る初期化処理です。
