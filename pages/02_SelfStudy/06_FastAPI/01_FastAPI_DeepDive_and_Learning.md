# 01. FastAPI 基礎理解から実践への道標と学習法

FastAPIの「本当の強み」を理解し、実務で戦えるレベルに到達するための学習ロードマップと、実践的な深掘り解説です。

## 1. 最短で実務レベルに到達する学習ステップ

初心者がいきなりデータベースに繋ぎに行くと、非同期処理（async/await）とSQLAlchemyの独特な書き方でほぼ確実に挫折します。以下の順序で学ぶのが最適解です。

1. **Step1: 公式チュートリアルを「読む」のではなく「写経」する**
   FastAPIの公式ドキュメントは、全フレームワークの中でも群を抜いて美しく、初心者に優しいことで有名です。「First Steps (最初のステップ)」から順に、必ず自分でタイピングして `localhost:8000/docs` （自動生成されるSwaggerドキュメント）がどう変化するかを体感してください。
2. **Step2: Pydanticモデルとの戦い**
   「クライアントからどんなJSONを受け取るか」「どんなJSONを返すか」を `BaseModel` をパズルゲームのように組み立てて定義する練習をします。ここで「型」のありがたみを理解します。
3. **Step3: Dependency Injection (DI) の理解**
   FastAPI最大の壁にして最高の武器である `Depends()` の概念を学びます。認証トークンのチェックなどを関数に注入する感覚を掴んでください。
4. **Step4: データベース (SQLAlchemy / SQLModel) との接続**
   最後にDB接続を行います。近年はFastAPIの作者が直接開発した `SQLModel` という、PydanticとSQLAlchemyを融合させた新技術を使うのが非常にモダンで簡単です。

## 2. 実践的深掘り：なぜ「非同期 (async)」なのか？

FastAPIを学ぶ上で避けて通れないのが `async def` という見慣れない構文です。
なぜただの `def` ではなく `async def` が必要なのか？

### 理由: I/Oバウンド（待ち時間）の打破

Web APIの仕事の9割は、「データベースからデータを探してくるのを**待つ**」「外部APIからデータが返ってくるのを**待つ**」という『待ち時間の発生（I/O待ち）』です。
同期通信（普通の `def`）の場合、この待ち時間中はサーバーが完全にフリーズし、他のユーザーからのアクセスを一切ブロックしてしまいます。

`async def` と `await` を使うと、**「あ、データベースくん今探してるのね。じゃあ君が結果を返してくれるまでの間、俺は別のお客さんのAPIリクエストをさばいてくるわ！」**と、サーバーが超高速で掛け持ち作業をしてくれます。これが非同期の絶大な力です。

> [!WARNING]
> 大いなる罠：`async def` の中で、通常の（同期的な）長い計算処理（CPUバウンド）や、非同期対応していない古いDBライブラリなどを書いてしまうと、せっかくの非同期の恩恵が破壊され、サーバー全体がフリーズします。迷ったら、逆に `async` を外して普通の `def` で書いた方が、FastAPI側が裏でうまいことマルチスレッドに逃してくれて安全に動く（FastAPIの賢い仕様）という超重要ルールがあります。

## 3. 実践的深掘り：強力すぎる「型」エコシステム

FastAPIは、Pythonの `Type Hints（型ヒント）` を極限まで利用して作られています。

```python
# id は必ず整数でなければならない！という強い意志
@app.get("/items/{id}")
async def read_item(id: int):
    return {"item_id": id}
```

実務におけるこの1行の破壊力は計り知れません。
もしフロントエンド（React等）のエンジニアが間違えて `GET /items/abc` と文字列を送ってきた場合、FastAPIは**私たちの書いた関数 `read_item` の中に入るより手前の段階で、自動的にフロント側に「422 Unprocessable Entity (エラー：整数を送ってください)」という完璧なエラーJSONを弾き返します**。

これにより、過去のFlask等で大量に書いていた `if type(id) is not int: return Error` という不毛なバリデーションコードが全滅し、開発スピードが数倍に跳ね上がるのです。
