# 00. FastAPIの実務概念と原点

## 1. 歴史と誕生背景

2018年、Sebastián Ramírezによって開発されました。
当時、Pythonのバックエンドといえば「超重量級のDjango」か「軽量だが型の概念がないFlask」の二択でした。

時代はすでにTypeScriptが浸透し「型がない言語はチーム開発でバーストする」という常識が生まれていた中、Pythonにも `Type Hints`（型ヒント機能）が導入されました。
Sebastiánは、「Flaskのように軽く、かつ最初から型推論（Pydantic）を前提とし、APIのドキュメント（Swagger UI）を完全自動生成する非同期フレームワーク」を目指し、FastAPIを生み出しました。

## 2. 概念と動作原理

### 非同期駆動 (ASGI) と高速性

FlaskやDjangoが長年採用していた「WSGI（同期通信）」という規格を捨て、非同期通信（`async/await`）に完全最適化された「ASGI (Asynchronous Server Gateway Interface)」規格を採用しています。
Node.jsやGo言語に匹敵するスピードを実現しており、AIなどの「裏で時間がかかるAPI」を提供する基盤として最強の適性を持っています。

### 型ヒント＝入力バリデーション＝APIドキュメント

FastAPIの最大の革命です。
関数にパラメーターを書くだけで、以下の3つが「完全に同時」に行われます。
1. エディタ上で型補完が効くようになる
2. 間違った型のデータが送信された際、自動で弾く（バリデーション）
3. そのAPIを使うための美しい仕様書サイト（Swagger UI）が勝手に生成される

## 3. 強みと弱み (Pros/Cons)

- **強み (Pros)**: 「APIサーバー」を作る用途において、2026年現在のPython界隈では完全なナンバーワンです。自動生成されるSwagger UIにより、フロントエンド（React/Next.js等）のエンジニアとの連携が異常なほどスムーズになります。
- **弱み (Cons)**: 「HTMLのWebページ（画面）」を返す昔ながらのサーバーサイドレンダリング（SSR）には適していません。また、比較的新しいフレームワークであるため、SQLAlchemy等の他ライブラリを非同期(async)で組み合わせる際に「非同期関数の罠」にハマりやすいという初学者のハードルがあります。

## 4. 本当の基礎事項

- **Pydanticモデルの絶対活用**
  FastAPIと一蓮托生である「Pydantic」というライブラリを使って、受け取るJSONの「型（モデル）」を作り込むのが全ての基本です。「idは必須の整数」「年齢は0〜120まで」といった制限をモデルにかけることで、不要な `if` 文による入力チェックを完全に排除します。
- **DI (依存性の注入: Dependency Injection) の存在**
  FastAPIの強烈な特徴です。「データベースの接続情報」や「共通の認証チェック処理」を、関数の引数に `Depends()` という形でポンと入れるだけで、裏側で勝手に処理して注入してくれます。これによりテストが極めて書きやすくなります。
