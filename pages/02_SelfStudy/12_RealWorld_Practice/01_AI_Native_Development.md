# AIネイティブ開発とエディタのパラダイムシフト

## 1. AI駆動型エディタとエンジニアの「判断・管理」

- **【辞書的な意味（公式定義）】**：従来型は人間が手入力し補完機能等で支援するソフトウェア。AI駆動型はLLMがコード生成・リファクタリング・デバッグなどを自律的または対話的に行い、人間がその指示や評価を担う開発パラダイムを前提としたエディタ（例：Cursor, Windsurf, GitHub Copilot Workspaceなど）。
- **【解説（細分化・噛み砕き）】**：従来のVS Codeなどは「超高機能なペンとノート」でした。いくら書きやすくても筆を動かすのは人間です。対してAIエディタは「超優秀だが少しおっちょこちょいな部下（ペアプロ相手）」です。これからのエンジニアは「タイピングする係」ではなく、「部下に要件を的確に伝え、上がってきたコードが仕様通りか、バグがないかを**判断・管理（レビュー）する係**」にシフトさせられます。
- **【現場でのリアルな意味・文脈】**：暗記力やコーディングの物理的な早さよりも、「要件を正確な言語（プロンプト）に落とし込む力」と「AIのコードの幻覚（嘘）を一発で見抜くコードレビュー能力」が最重要視されます。
- **【ゲームに例えると】**：FF14の「フェイス（NPCとダンジョンに行くシステム）」に近いです。今までソロで全アクションを手動入力していたのが、優秀なNPCに大半を任せられるようになりました。ただし、NPCがギミック処理をミスらないように、あなたが「どこに移動すべきか」「今使うべきスキルは何か」を的確に指示・管理する必要があります。
- **【実務でのトレードオフ】**：
  - *メリット*: 開発速度が理論上数倍になり、定型文の記述や周辺技術の学習コストが激減します。
  - *デメリット*: AIが生成したブラックボックスなコードを「よく分からないけど動いたからヨシ！」で放置すると、後に誰も直せない致命的なバグの温床になります。

## 2. システムプロンプト設定ファイルの実践構築

AIにプロダクトのルールを教え込む「システムプロンプト設定ファイル」は、現場でのAIの品質を左右する最重要ファイルです。

### ツール別ルールファイルの命名規則と適用範囲

AIエディタ（自律型エージェント）によって、プロジェクト内に配置すべき設定ファイルやディレクトリの構造が異なります。
- **Cursor**: `.cursorrules` (プロジェクト全体に適用される単一の基本ルールファイル)
- **Windsurf**: `.windsurfrules` (Cascade機能などに適用される単一のルールファイル)
- **Antigravity (まさに私です)**: `.agent/skills/<スキル名>/SKILL.md` (用途ごとに「スキル」として複数フォルダに分割・構造化し、AIがタスク状況に応じて必要なスキルを動的に読み込む高度な方式)
- **GitHub Copilot**: `copilot-instructions.md` 等（環境によって設定場所が異なる場合あり）

※ これらは Git で管理し、チーム全体で品質を均一化するのがベストプラクティスです。

### ルールファイル構築のベストプラクティス（記述手法）

AIに曖昧な指示を出すと推測で間違ったコードを書くため、以下のように「マークダウンの箇条書き」で明確に構造化して記述します。

#### ① ペルソナ（ロール）の定義

AIがどういう立場で回答すべきかを最初に定義します。

```markdown
# Role
あなたはシニアレベルのフロントエンドエンジニアであり、React/Next.jsの最新のベストプラクティスに精通しています。
```

#### ② 技術スタックとバージョンの「完全固定」

バージョンを指定しないと、AIはネット上の古い記事を参照し、非推奨（Deprecated）な書き方を頻発します。

```markdown
# Tech Stack
- Next.js (App Router必須, Pages Routerは使用禁止) v14
- React v18
- Tailwind CSS v3
```

#### ③ やってはいけない事（Negative Prompt）

AIは気を利かせて勝手なことをしがちです。「〜しないでください」という禁止事項を明記します。

```markdown
# Constraints (禁止事項)
- 勝手に既存のコメントやDocstringを削除しないでください。
- ユーザーに確認せずに、破壊的なコマンド（rm -rf 等）や依存関係のインストール（npm install）を実行しないでください。
- `any` 型は使用せず、必ずZod等でスキーマ定義を行ってください。
```

#### ④ ディレクトリ構造のインデックス化

AIにプロジェクトのどこに何があるかを教えるため、簡単なディレクトリマップを記載しておくと、AIが迷子になりません。

```markdown
# Directory Structure
- `/components/ui`: 汎用的なUIコンポーネント
- `/app/api`: サーバーサイドのAPIルート
```

- **【実際の会話での使われ方】**：「AIがまた古いReactの記法（Classコンポーネント等）でコード吐いてきたから、`.cursorrules` の技術スタックの項目に『Functional Componentのみ使用』って追記しておいて。」
