# エンジニア生存戦略と行動規範

## 1. 15分ルール (15-Minute Rule) と質問の作法

- **【公式定義】**：問題に直面した際、最初の15分間は自力で解決策を調べ、それでも解決の糸口が見えない場合は、一人で抱え込まずに必ずチームメンバーやシニアに助けを求めるというタイムボックス型のルール。
- **【現場でのリアルな意味】**：新人が丸1日かけても終わらないエラーは、シニアがアーキテクチャの前提を1分確認するだけで終わることが多々あります。プロジェクトの納期と予算を守るためのリスク管理手法です。
- **【ゲームに例えると】**：FF14の高難易度レイドでの「ギミックの相談」です。ワイプした原因を勝手に抱え込んで無言でリトライするのではなく、「15分間様子見しましたけど原因わからなかったので、タンクさん軽減の相談させてください」と早めに聞き、パーティ全体の時間を無駄にしない行動と同じです。

### 評価される「質問テンプレート」

「調べた過程の共有」をせずに雑な質問（「動きません。どうすればいいですか？」）をすると、シニアからの信頼を著しく損ないます。以下の3点セットを必ず添えるのがプロの作法です。

```markdown
【やりたい事（目的）】
- ユーザー登録処理の際に、Zodでバリデーションを行いたい。

【発生している事象（現状）】
- `npm run dev` 実行時に `ZodError: Invalid email` というエラーが出て画面が白くなる。

【試したこと・自分なりの推測】
1. コンソールログを出力した結果、フォームの入力値自体は取得できていた。
2. データベースの接続は問題ないことを確認済み。
3. 推測ですが、Zodのスキーマ定義で必須（Required）にしている項目に、フロントからUndefinedが渡っているのではないかと考えています。

ここから先のデバッグ方法についてアドバイスいただけないでしょうか？
```

これを見せられたシニアは「おっ、こいつは問題の切り分けができているな」と評価し、喜んで数分で解決策を提示してくれます。

## 2. AI時代における「防衛的プログラミング」の復権

- **【公式定義】**：予期せぬ入力（Nullや異常値）が発生しても、システム全体が破綻しないよう事前にバリデーションや例外処理を徹底する設計手法。
- **【現場でのリアルな意味】**：AIは「その場しのぎで動くそれっぽいコード」を超高速で生み出します。しかし、エラー時の処理が甘かったり、構造が脆弱だったりすることが多々あります。「爆速でAIが大量生産してくるコードが、後々システムを破壊するゴミにならないよう、人間側が強いフィルターとなる」ために、これらの設計知識がいま猛烈に再評価されています。

### 防衛的実装の具体例：早期リターン（Guard clauses）

AIが適当に書きがちな深くネストされたIF文（アンチパターン）を防ぎ、可読性と安全性を高める必須の手法です。

**❌ AIが書きがちな悪い例（ネストが深い）**

```javascript
function processUser(user) {
  if (user != null) {
    if (user.isActive) {
      if (user.hasPaymentMethod) {
        // メインの処理がこんなに深い場所に...
        chargeUser(user);
      } else {
        throw new Error("支払い方法がありません");
      }
    } else {
      throw new Error("アクティブなユーザーではありません");
    }
  } else {
    throw new Error("ユーザーが存在しません");
  }
}
```

**✅ 防衛的で美しい例（早期リターン）**

```javascript
function processUser(user) {
  // ① 異常系（ダメな条件）を最初に全て弾いて処理を抜ける
  if (user == null) throw new Error("ユーザーが存在しません");
  if (!user.isActive) throw new Error("アクティブなユーザーではありません");
  if (!user.hasPaymentMethod) throw new Error("支払い方法がありません");

  // ② ここまで到達した変数は「100%安全」が保証されている。メイン処理を浅い階層で書ける
  chargeUser(user);
}
```

「AIが吐き出したこの関数、たしかに動くけど防衛的じゃないね。早期リターンを追加するよう、プロンプトを打ち直して。」というのがシニアの視点です。
