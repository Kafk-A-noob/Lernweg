# モダンインフラの解読（K8s / GitOps等）

## 1. Kubernetes (K8s) とコンテナオーケストレーション

- **【辞書的な意味（公式定義）】**：コンテナ化されたアプリケーションのデプロイ、スケーリング、ネットワーク管理を自動化するためのオープンソースシステム。
- **【解説（細分化・噛み砕き）】**：1台のサーバーですべてをまかなうシステム構成が「店主が1人で調理からレジまで全対応する定食屋」だとすると、K8sは「超大型チェーンの自動化ファミレス」です。お客の数（トラフィック）が増えたら自動でロボット店員（コンテナ）の数を増やし、店員が壊れたら即座に処分して新品の店員を補充します。この店員を統括する「鬼の店長」システムがK8sです。
- **【現場でのリアルな意味・文脈】**：サーバーに対する考え方が「ペット（名前をつけて大事に直しながら育てる）」から「家畜（壊れたら即座に入れ替えるコンテナ群）」へ根本から変わります。障害復旧を「人間が治す」のではなく「死んだコンテナを捨てて、新品を自動的に立てる」というアプローチです。
- **【ゲームに例えると】**：マイクラの「Mekanism等の高度な物流パイプライン＋自動クラフト施設」です。1台の手動かまどではなく、アイテムが来たら自動で複数のかまど（Pod）に分配し、壊れたら自動修復されるような巨大工場を構築するのと同じです。
- **【実務でのトレードオフ】**：
  - *メリット*: 大規模なアクセス増減に自動対応し、理論上システムが非常に止まりにくくなります（自己修復・高可用性）。
  - *デメリット*: 仕組みが複雑怪奇になりすぎるため、個人レベルの小さなアプリにはいわゆる「牛刀をもって鶏を割く（明らかなオーバースペック）」状態になります。
- **【実際の会話での使われ方】**：「アクセス急増でPodが過負荷で落ちてるけど、K8sのレプリカセットが自動で再起動かけてくれてるから、とりあえずは原因のログだけ追っておいて。」

## 2. K8s環境の実務的な読み解き順（外から中へ）

- **【辞書的な意味（公式定義）】**：アプリケーションへの外部からの通信を、内部のコンテナ群（Pod）へ適切に中継・ルーティングするためのリソース（Ingress, Service, Pod等）の依存関係順序。
- **【解説（細分化・噛み砕き）】**：初学者がいきなり中身のプログラムから読もうとすると全体像を見失います。K8sを読む際のベストプラクティスは「外から中へ（ユーザーのアクセス経路）」で追うことです。
  1. `Ingress` (大門の門番): どのドメイン・URLで来た客を中に入れるか？
  2. `Service` (受付係・バランサー): 入ってきた客を、裏側のどの部屋（コンテナ群）に割り振るか？
  3. `Deployment / Pod` (実働部隊): そこにどんなアプリの本体（イメージ）が、何個立ち上がっているか？
- **【現場でのリアルな意味・文脈】**：画面が映らない（502 Bad Gateway等）エラーが出た際、この「道筋」のどこで通信が途絶えているかを見極めるのがインフラ・運用の最初の仕事です。
- **【ゲームに例えると】**：マイクラのパイプライン構築で、「アイテムの搬入口（Ingress）」をまず探し、次に「パイプの分岐器・仕分け機（Service）」の行き先を追い、最後に「精錬かまど群（Pod）」を確認して、どこでアイテムが詰まっているかを特定するトラブルシューティング手順と同じです。
- **【実務でのトレードオフ】**：
  - *メリット*: この階層構造があるおかげで、中身のPodを無停止で丸ごと入れ替えたり、通信の振り分けルールだけを後から変更したりするのが容易になります。
  - *デメリット*: どこか一つでも設定（ラベルの結びつけ等）を書き間違えると、たちまち全体が通信不通になる脆さがあります。
- **【実際の会話での使われ方】**：「Serviceまでは疎通してるみたいだから、そこに紐づいている背後のPodかDeploymentのラベル設定ミスってないか確認して。」

## 3. GitOps と Argo CD

- **【辞書的な意味（公式定義）】**：Gitリポジトリを「システムの状態の唯一の情報源（Single Source of Truth）」とし、Gitへの統合（マージ）をトリガーに、本番環境の状態を宣言的に同期させる運用手法。
- **【解説（細分化・噛み砕き）】**：昔は本番環境を変えたい時、エンジニアが黒い画面でサーバーに入り、直接コマンドを叩いていました（手オペ）。GitOpsでは「Git上の設計図を書き換えるだけ」です。Argo CDという裏方ボットが、「Gitの設計図」と「いまの本番環境」を見比べて、「あれ、本番が設計図と違うぞ？」と気づき、本番環境を自動的に設計図通りに組み替えてくれます。
- **【現場でのリアルな意味・文脈】**：人間が直接本番のサーバーを操作するリスクを排除する仕組みです。万が一エラーが起きても、「誰が、いつ、何をミスしたか」がGitの履歴にすべて残り、「1つ前のコミットに戻す」だけで本番環境の崩壊も元に戻るという極めて高い回復力を持ちます。
- **【ゲームに例えると】**：Unity開発における「Prefabの変更」やマイクラの「WorldEdit設計図」です。あなたがGit上の設計図を変更すると、ArgoCDという監視ボットが、稼働中の本番ワールド内にある全ての実体オブジェクトを自動で最新形に一斉更新してくれます。
- **【実務でのトレードオフ】**：
  - *メリット*: 本番を手動で壊す人的ミスが激減し、誰が環境を変えたかの証跡（履歴）が確実に保持されます。
  - *デメリット*: Gitの設計図の記述自体にエラーを含んだままPushしてしまうと、ArgoCDが律儀にそれを読み取り、本番環境が一瞬で崩壊する恐れがあります。
- **【実際の会話での使われ方】**：「本番のリリース、Argo CDで同期されるよう組んでいるから、マニフェストファイルを書き換えてmainブランチにマージするだけで勝手にデプロイされるよ。」
