# FlaskとSQLAlchemyによる論理削除と表示一貫性の設計

データベースを伴うWebアプリケーションにおいて、データの削除をいかに安全に扱うかはシステム設計の要となる。本稿では、物理削除に伴うリスクとその解決策である論理削除の概念、およびFlask-SQLAlchemyにおける実装のベストプラクティスを解説する。

---

## 1. 物理削除と論理削除の概要とジレンマ

### 物理削除 (Physical Delete / Hard Delete)

データベースのテーブルから、レコードを完全に消去する手法 (SQLの`DELETE`文)。

- **メリット**: データベースのストレージ容量を節約でき、検索速度の低下を防ぐことができる。
- **デメリット**: 誤操作時に復旧が極めて困難 (バックアップのリストアが必要)。また、関連する過去のデータ (例：退会したユーザーの購入履歴など) との整合性が破壊され、システムエラーの原因となる。

### 論理削除 (Logical Delete / Soft Delete)

レコード自体はデータベース上に保持したまま、専用の削除フラグ (`del_flag` 等) カラムを更新し、システム上は消えたこととして扱う手法 (SQLの`UPDATE`文)。

- **メリット**: 誤操作時の復旧が容易。過去の統計データや関連データとの整合性を保ったまま安全に運用できるため、**現代の商用システムにおいては論理削除が標準的に採用される**。
- **デメリット**: データベースの実容量が増え続ける。また、表示する際に毎回『削除されていないものだけ』という検索条件を付け忘れると、削除済みのデータが画面に露出するという実装上の罠が存在する。

---

## 2. SQLAlchemyにおける論理削除の実装

Flask-SQLAlchemyを用いて論理削除を実装する場合、削除ルーティングにおいて `db.session.delete()` を呼ぶのではなく、対象インスタンスのフラグプロパティを書き換えて `commit()` する。

### ルーティング側の修正 (DeleteからUpdateへ)

```python
# 物理削除(非推奨)の場合
# db.session.delete(emp)
# db.session.commit()

# 論理削除の場合
emp = Employee.query.get_or_404(id)
# 1対1や1対多で紐づく親テーブル(Person)がある場合は、連動して論理削除状態にする
person = emp.person

emp.del_flag = 1
if person:
    person.del_flag = 1

db.session.commit() # UPDATEとして反映される
```

このように、モデルの設計段階で各テーブルに連携用の `del_flag` を用意しておくことが前提となる。

## 3. 最も陥りやすい罠: 表示系クエリへのフィルター適用

論理削除を採用したシステムにおいて最も多発するバグは、**消したはずのデータがセレクトボックスの選択肢に表示されてしまう存在しないはずのユーザー情報にURL直叩きでアクセスできてしまう**といった、表示・検索条件の適用漏れである。

これらを防ぐため、データを取得する (`SELECT`する) すべての箇所で厳密なフィルタリングを行う必要がある。

### 単純な `!= 1` の罠と `None` の安全なハンドリング

```python
# 変更前 (すべてのデータが取得されてしまう)
data = Employee.query.all()

# 変更後 (論理削除済みのデータを除外するフィルター処理)
data = Employee.query.filter((Employee.del_flag == None) | (Employee.del_flag != 1)).all()
```

#### なぜ `(Employee.del_flag == None)` の条件も加えるのか？

SQLiteを含め、多くのリレーショナルデータベースでは、初期作成時 (レコード生成時) に `del_flag` に値が明示的にセットされていない場合、`NULL` (Python環境における `None`) という空の状態が入っている。

SQLの仕様上、**`NULL != 1` という比較演算はTrue (真)ではなくNULL (不明)と判定される**。
そのため、1ではないものという単一の条件 (`Employee.del_flag != 1`) だけで抽出すると、初期状態のままの健全なレコードまで抽出から漏れて画面から消えてしまうという重大なバグを引き起こす。

これを防ぐための安全対策 (フォールバック) として、以下のように論理和(`|`)を用いた厳密な条件式を `filter()` に記述する必要がある。

- `Employee.del_flag == None` : フラグが未設定 (初期状態) のもの
- `Employee.del_flag != 1` : 何かしら値が入っているが、1 (削除済) ではないもの

この設計思想は、単なるWebフレームワークの知識を超え、エンタープライズ領域におけるデータ整合性担保の基礎となる。
