# DB設計とアーキテクチャのお作法 (DB設計・インデックス・トランザクション)

「とりあえずテーブルを作ってデータを保存できれば良い」という発想は、運用フェーズに入った瞬間に崩壊する。データベース設計における実務上の原理原則と、安全にシステムを運用するためのトランザクションの概念を解説する。

---

## 1. 論理削除 vs 物理削除 の設計論

Flaskの学習課題 (P04・P05) でも扱った通り、システムのデータ削除要件には2つのアプローチがある。

### 物理削除 (Hard Delete)

- **手法**: `DELETE FROM table;` で完全に消滅させる。
- **採用ケース**: GDPRなどの法令で「退会ユーザーのデータを完全に消すこと」が義務付けられている場合。あるいは、ログデータなど不要でディスクを圧迫するもの。

### 論理削除 (Soft Delete)

- **手法**: `UPDATE table SET del_flag = 1;`
- **採用ケース**: 現代の商用システムの9割。過去データの統計分析や、誤操作からの復旧担保のため。
- **DB設計のお作法**: 論理削除を採用する場合、各テーブルの `del_flag` と併せて、いつ誰が消したのかの証拠跡を残すため `deleted_at` と `deleted_by` というタイムスタンプとユーザーID用カラムを用意するのがエンタープライズの鉄則である。

## 2. データの正規化 (Normalization) と 非正規化

- **公式リファレンス**: [データベース正規化の基礎 - Microsoft Docs](https://learn.microsoft.com/ja-jp/office/troubleshoot/access/database-normalization-description)

DB設計を行う際、データをどのように分割 (テーブル分け) するかの基準となるのが「正規化」である。

### 正規化の目的 (DRY原則)

「データの重複を無くし、更新時の不整合を防ぐこと」。
例えば、ユーザー情報の中に「神奈川県横浜市...」と毎回ベタ書きしていると、もし「横浜市」の名前が変わった時に全ユーザーのデータを書き換えなければならなくなる。
そこで、「都道府県マスタ」という別テーブルを作り、ユーザー側には「ID: 14 (神奈川)」という番号だけを持たせる。これが**正規化**である。

### 非正規化 (Denormalization) のジレンマ

正規化をやりすぎると、「1人のユーザーの画面を出すために、都道府県、性別、部署、役職...と10個以上のテーブルをすべて `JOIN` しなければならない」という事態に陥り、検索速度 (パフォーマンス) が著しく悪化する。

- **お作法**: 基本は正規化を優先し、著しく速度が出ない・計算が重すぎる部分に限り、あえて重複データを持たせるなどの「非正規化」を**戦略的に**行う。

## 3. インデックス (Index) の両刃の剣

- **公式リファレンス (PostgreSQL)**: [インデックス - PostgreSQL Docs](https://www.postgresql.jp/document/current/html/indexes.html)

インデックス (索引) とは、データベースの検索速度を劇的に跳ね上げるための機能である。
辞書の「あ」「い」「う」のツメ (インデックス) と同じく、これが貼られているカラムは爆速で `SELECT` (検索) できるようになる。

### 【罠】 インデックスの貼りすぎによる崩壊

「検索が早くなるなら全部のカラムに貼ればいいのでは？」と思うかもしれないが、これは絶対にしてはいけない。

- **理由**: インデックスを貼ると、データが追加 (`INSERT`) や更新 (`UPDATE`) されるたびに、データベースは**裏で「索引 (ツリー構造) 」も書き直さなければならない作業**が発生する。つまり、`SELECT`が早くなる代わりに、`INSERT`や`UPDATE`が激重になるトレードオフの関係がある。
- **お作法**: インデックスは、`WHERE` 句での絞り込みや `ORDER BY`、テーブル間の結合 (`JOIN` のキー) で**頻繁に使われるカラムにのみ**厳選して貼るのが鉄則。

## 4. トランザクション (Transaction) と ロック

- **公式リファレンス**: [トランザクション制御 - PostgreSQL Docs](https://www.postgresql.jp/document/current/html/tutorial-transactions.html)

「銀行の口座振替」などで不可分な操作を保証する仕組み。

### ACID特性の保証

Aさんの口座から1万円を引き (処理1)、Bさんの口座に1万円を足す (処理2)。
もし処理1の後にサーバーの電源が落ちたら？ 1万円は電子の海に消滅してしまう。
これを防ぐのがトランザクションである。

```sql
BEGIN; -- トランザクション開始
UPDATE accounts SET balance = balance - 10000 WHERE name = 'A';
UPDATE accounts SET balance = balance + 10000 WHERE name = 'B';
COMMIT; -- 処理が【全て】成功したら確定させる
-- ※もし途中でエラーが起きたら ROLLBACK; を呼ぶことで、「処理1」も含めて全て無かったことになる。
```

### 排他制御 (ロック)

あるプログラムがトランザクション中でAさんのデータを見ている間、別のプログラムが同時にAさんのデータを書き換えようとするのを防ぐ (ロックする) 仕組み。

- **悲観的ロック**: 「絶対に誰かが同時に触りに来る」と想定し、見始めた瞬間に他者を完全にブロックする。
- **楽観的ロック**: 「おそらく誰も触らないだろう」と想定し、保存する最後の瞬間に「自分がデータを取得してから今までに、誰か更新した？」とバージョン (`version`カラムや`updated_at`) をチェックする仕組み。近年のWebサービスではこちらが主流。
