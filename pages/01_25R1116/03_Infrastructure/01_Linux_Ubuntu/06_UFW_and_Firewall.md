# 06. UFW (ファイアウォール) とポートセキュリティ

Linuxサーバーを外部ネットワーク（インターネット）に公開した瞬間、世界中の悪意あるアクセスがポートを叩きにやってきます。
UFW（Uncomplicated Firewall）は、Ubuntu標準のファイアウォール管理ツールであり、**「どのポート（玄関のドア）を開けるか／閉じるか」を極めてシンプルなコマンドで制御する門番**です。

## 1. そもそもファイアウォールとは何か？

サーバーは「ポート番号」という番号付きの扉を持っています（全部で65535個）。
例えばWebサーバー（Apache/Nginx）は80番と443番の扉を開けて、来訪者（HTTPリクエスト）を待ち受けます。SSHは22番ポートです。

ファイアウォールとは、**「許可した特定の扉以外は、すべて施錠して外部からの侵入を拒否する」**ためのセキュリティの壁です。

### なぜ必要なのか？

ファイアウォールが無い状態は、「家のすべてのドアと窓が開いている状態」と同じです。
仮にSSHサーバーが22番で起動していなくても、データベース（MySQL: 3306番）が意図せず外部に公開されていたりすると、そこから侵入されてデータを抜かれます。必要なポート以外は「デフォルトで全閉」にするのがセキュリティの大原則です。

## 2. UFWの概念と動作原理

UFWの裏側では、Linux カーネルレベルの強力なパケットフィルター「**iptables (netfilter)**」が動いています。
しかし、iptablesの設定コマンドは長くて非常に難解なため、それを人間フレンドリーにラップした「UFW（Uncomplicated = ややこしくない Firewall）」が生まれました。

### デフォルトポリシーの考え方

UFWの鉄則は「**デフォルト拒否（deny incoming）**」です。
まずすべてのポート（入り口）を閉じた上で、必要な通信だけをピンポイントで許可（allow）します。これを「ホワイトリスト方式」と呼びます。

## 3. 実践：UFWの基本操作

### 初期設定（まず最初に必ず行うこと）

```bash
# UFWの現在の状態を確認する（有効/無効、ルール一覧）
sudo ufw status verbose

# デフォルトポリシーの設定：「外から来る通信は全て拒否」「中から出る通信は全て許可」
sudo ufw default deny incoming
sudo ufw default allow outgoing
```

> ⚠️ **超重要な注意**: SSHポート（22番）を許可する前にUFWを有効化すると、**自分自身がサーバーに二度とログインできなくなります（完全締め出し）**。必ず先にSSHを許可してからUFWを有効化してください。

### ポートの許可と拒否

```bash
# SSH (22番) を許可する（これが最初の1コマンド）
sudo ufw allow ssh
# もしくは番号で指定
sudo ufw allow 22/tcp

# HTTP (80番) と HTTPS (443番) を許可する
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# もしくはサービス名のプロファイルで一括指定
sudo ufw allow 'Nginx Full'   # 80と443が同時に許可される
sudo ufw allow 'Apache Full'

# 特定のポートを拒否する（閉じる）
sudo ufw deny 3306/tcp  # MySQLポートを外部からの接続禁止にする
```

### UFWの有効化と無効化

```bash
# UFWを有効化する（再起動後も自動的にルールが適用される）
sudo ufw enable

# UFWを一時的に無効化する
sudo ufw disable

# ルールを番号を指定して削除する
sudo ufw status numbered  # まず番号を確認
sudo ufw delete 3         # 3番のルールを削除
```

### 特定IPアドレスからの接続のみを許可する

最も堅牢な設定です。「社内ネットワークのIPからだけSSHログインを許可する」といったアクセス制限に使います。

```bash
# 特定のIP(192.168.1.100)からのSSH接続のみを許可する
sudo ufw allow from 192.168.1.100 to any port 22

# 特定のサブネット(10.0.0.0/24)全体からのアクセスを許可する
sudo ufw allow from 10.0.0.0/24
```

## 4. 実務でのアンチパターンと失敗例

- ❌ **UFWを有効化したのにSSHの許可を追加し忘れ、自分自身が締め出される**
  - クラウド環境（AWS EC2等）でこれをやると、物理的なコンソールアクセス手段がない限りインスタンスを破棄（terminate）して作り直す羽目になります。ローカルVMやWSLでの練習段階で「先にSSH、次にenable」の手順を身体に叩き込んでください。
- ❌ **「とりあえず `ufw allow` を連打して全ポートを開放する」**
  - これはファイアウォールを設定していないのと全く同じ状態です。開けたポートは全て攻撃対象になります。

## 5. 2026年最新のプラクティス

1. **クラウド環境ではUFWとクラウドSGの二重防御が必須**
   AWSの「Security Group」やGCPの「Firewall Rules」は、UFWより手前のネットワークレイヤーで動きます。クラウド上の本番環境では、クラウド側のSGで粗くフィルタリングし、OS内部のUFWで細かく制御する二段構えが理想です。
2. **`ufw logging` によるアクセスログの監視**
   `sudo ufw logging on` を有効にしておくことで、拒否（ブロック）された通信のログが `/var/log/ufw.log` に記録されます。不正アクセスの試行を検知し、該当IPを特定してブラックリスト化する（`fail2ban` との連携）のが運用現場の定石です。
