# 04. パーミッション(chmod)と権限

Linux上の全てのファイルとディレクトリには、「誰が」「何をしていいか」という強固なアクセス権（パーミッション）が設定されています。

## 1. 権限の基本 (r / w / x)

権限には以下の3つの種類があり、それぞれに「数字」が割り当てられています。

- `r` (Read: 読み取り) = **4**
- `w` (Write: 書き込み) = **2**
- `x` (eXecute: 実行) = **1**

### なぜ 4, 2, 1 なのか？（重要）

これを足し算することで、「どんな組み合わせでも一意の数字になる」からです。
- 読めて(4)＋書ける(2) ＝ **6**
- 読めて(4)＋書けて(2)＋実行できる(1) ＝ **7**
- 読むこと(4)しかできない ＝ **4**

## 2. 三つのレイヤー（所有者・グループ・その他）

一つのファイルに対して、権限を与える対象は3つの属性に分かれています。

1. **所有者 (U: User)**: このファイルを作った人
2. **グループ (G: Group)**: このファイルの所有グループに属している人たち
3. **その他 (O: Other)**: まったく関係ない赤の他人

### chmodによる権限の付与

`chmod 744 test.txt` というコマンドがあった場合、上の数字の足し算を「3桁」並べているだけです。

- 1桁目 (`7`): 所有者は 読めて書けて実行できる (4+2+1)
- 2桁目 (`4`): グループは 読むことしかできない
- 3桁目 (`4`): その他の他人は 読むことしかできない

## 3. Web実務・LAMP環境における意味

WordPressなどのPHPシステムを自分で構築する際、必ずと言っていいほどパーミッション変更（`chmod 777` や `chmod 755`）が必要な場面に直面します。

**よくある実務のトラップ：**
画像のアップロード機能を作ったのに、サーバーに画像が保存されずエラーになることがあります。
これは、Apache（Webサーバー/プログラム）は内部的に「システムユーザー（`www-data`などが該当）」として動いており、アップロード先のフォルダに対して**「その他の他人（Other）も書き込み可能(`w: 2`)」な権限**が与えられていないと、システムが画像ファイルを書き込めずハジかれてしまうからです。

そのため実務では、Webアプリが書き込む必要がある特定の一部のディレクトリにだけ `chmod 777` や `707` 等を用いたり、サーバープロセスを特定のグループに所属させ、グループ権限(`chmod 775`等)のみで制御するなどの高度なセキュリティ設定が行われます。

---

## 4. 実務でのアンチパターンと失敗例

- ❌ **エラー解決のために脊髄反射で全てを `chmod -R 777` にする**
  - 動かないからといってルートフォルダ丸ごとに `777` (誰でも読み書き実行可能) を付与するのは「家のドアを全て取り外す」のと同じです。マルウェアの標的になり、一瞬でサーバーが乗っ取られる（CryptoMiner等を仕込まれる）最悪のアンチパターンです。
- ❌ **実行権限 (`x`) をむやみにファイルに付与する**
  - 単なる画像やテキストファイルに `x` 権限が付いていると、万が一ウイルスファイルがアップロードされた際にサーバーがそれを「プログラムとして実行」できてしまい、致命的なバックドアを開きます。通常のファイルは絶対に `644` (rw-r--r--) に留めるべきです。

## 5. 2026年最新のベストプラクティス

1. **ディレクトリは `755`、ファイルは `644` を徹底する**
   CMS（WordPressなど）やWebサーバーにおけるデファクトスタンダードの権限です。
   所有者のみフルアクセス、他の人は「読むだけ」というミニマムアクセスを担保します。
2. **パーミッションよりも「所有者（chown）」の最適化**
   無理に `777` にするのではなく、ファイルの所有者自体をプログラムの実行ユーザー（例：`www-data` や `nginx`）に変更（`chown`）することで、安全に `755` のまま書き込みを許可するアプローチが好まれます。
3. **コンテナ化による権限問題の隔離**
   現代の開発では、Dockerコンテナ内の独立したroot権限でプログラムを動かすため、ホスト側のパーミッションで悩むこと自体が減っています（コンテナごとの隔離によってセキュリティを担保するパラダイム）。

## 6. トラブルシューティング（よくあるエラー）

- **Q. `bash: ./script.sh: Permission denied` （スクリプトが実行できない）**
  - **A.** ファイルに実行権限(`x`)がありません。`chmod +x script.sh` (または `chmod 755 script.sh`) で実行権限を与えてください。
- **Q. WSLとWindows間でファイルのパーミッションがおかしくなった (全部777になる等)**
  - **A.** Windows(NTFS)とLinux(ext4)のファイルシステムの互換性の問題です。`/etc/wsl.conf` に `[automount] options = "metadata"` を設定することで、Windows側のマウント領域でもLinux本来のパーミッションを保持・変更できるようになります。

## 7. 頻出メソッド・必須コマンド大全

ファイルの権限や所有者を変更するための必須コマンド群（暗記推奨）。

```bash
# 所有者に実行権限だけをサクッと追加する（非常によく使う）
chmod +x filename.sh

# ディレクトリは755、内部のファイルは644に一括で設定する（WordPress等の復旧で使う魔法の呪文）
find ./ -type d -exec chmod 755 {} \;
find ./ -type f -exec chmod 644 {} \;

# ファイルの「所有者」と「グループ」を両方変更する
# 書式: sudo chown 所有者:グループ ファイル名
sudo chown www-data:www-data /var/www/html/uploads

# 再帰的（フォルダ配下の全ファイル）に所有者を変更する
sudo chown -R ubuntu:ubuntu /home/ubuntu/project
```
