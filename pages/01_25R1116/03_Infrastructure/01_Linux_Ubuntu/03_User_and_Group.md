# 03. ユーザー・グループ管理とsudo

Linuxは「完全に隔離された複数人のユーザーが同時に使う」ことを前提（マルチユーザーシステム）とした強固なセキュリティ設計を持っています。

## 1. 最重要ファイル： /etc/passwd と /etc/group

Linuxの世界では、ユーザーのリストなどはすべて「単なるテキストファイル」として保存・管理されています。

### `cat /etc/passwd`

システムに存在する全てのユーザーの情報が書かれています。
**書式:** `ユーザー名:パスワード(xで隠蔽):UID:GID:コメント:ホームディレクトリ:ログインシェル`

### `cat /etc/group`

すべてのグループの情報が書かれています。
**書式:** `グループ名:パスワード(x):GID:所属しているユーザーのリスト（カンマ区切り）`

## 2. UIDの概念（Linuxの階級システム）

システムは人間を「名前」ではなく「背番号 (UID: User ID)」で識別します。

- **`0`**: `root` ユーザー。神です。何でもできます。（Windowsでいう完全なAdministrator）
- **`1 - 999`**: システムユーザー。Webサーバー(`www-data`など)やDBなど、バックグラウンドのプログラムが「人間ではないが権限を分離するために使う」幽霊アカウント群です。
- **`1000 -`**: 一般ユーザー。人間がログインして使うためのアカウントです。

## 3. rootになる (sudo su -)

一般ユーザーは重要な設定ファイルを編集できません。神である `root` に変身するか、または `root` の権限を一時的に借りる必要があります。

- `sudo <実行したいコマンド>`: 今回だけ `root` の権限を借りて実行する。
- `sudo su -`: 今から自分自身が完全に `root` ユーザーへと変身（スイッチ）する。（環境変数等のプロファイルも全てrootのものに切り替わります）

## 4. ユーザーの追加と「sudo」権限の付与

授業のログにあった `student` ユーザーの関連操作です。

### ユーザーの新規作成と削除

```bash
# student というユーザーを新しく作る（ホームディレクトリも作られます）
sudo adduser student

# パスワードを変更する
sudo passwd student

# studentユーザーを削除する（ホームディレクトリのデータごと完全に消す）
sudo deluser --remove-home student
```

### グループへの追加（usermod）

一般ユーザーである `student` は、初期状態では `sudo`（管理者権限を借りる権利）を持っていません。
Linux（Ubuntu）では、**「`sudo` という名前のグループに所属している人物だけが `sudo` コマンドを使える」**というルールがあります。

よって、`student` に管理者権限を与えるには以下のコマンドを打ちます。

```bash
# -aG (アペンド・グループ): student を sudoグループ に追加する
sudo usermod -aG sudo student

# studentが今どのグループに入っているかを確認する
groups student
```

## 5. 実務でのアンチパターンと失敗例

- ❌ **`usermod -G`（`-a`なし）でグループを追加してしまう**
  - `-a` (append) を付け忘れると、「指定したグループ**だけ**に所属する」ように上書きされ、元々入っていた他の重要なグループ（`adm` や `docker` など）から全て脱退させられてしまいます。システムが破壊される原因となる、最恐級のアンチパターンです。
- ❌ **何でもかんでも `root` ユーザーで作業する**
  - 常に `sudo su -` で `root` になりっぱなしで作業すると、誤ってシステム破壊コマンド（`rm -rf /`等）を打った瞬間にPCが終わります。「必要な時だけ一般ユーザーから `sudo` を使う」のが鉄則です。

## 6. 2026年最新のベストプラクティス

現代のインフラ管理におけるユーザー管理の常識です。

1. **パスワード認証の廃止と公開鍵認証の徹底**
   実務のサーバーでは、パスワード（`sudo passwd`）を使ったログインはブルートフォース攻撃の餌食になるため無効化されています。必ず「SSH公開鍵認証」によるログインのみを許可する設定(`PasswordAuthentication no`)を行います。
2. **`sudoers` の細やかな権限分離**
   インフラエンジニア以外の開発者には、「再起動コマンドだけは `sudo` なしで打てるが、他の管理者権限は持たない」のように `/etc/sudoers` を編集し、最小権限の原則（Least Privilege）を適用します。
3. **IaC (Infrastructure as Code) によるユーザー管理**
   ユーザー作成コマンド（`adduser`）を手動で打つ行為自体が減っています。Ansible、Terraform、Docker（Dockerfile）などを使って、「コードを実行したら初期ユーザーが全部出来上がっている」状態を自動化するのがベストプラクティスです。

## 7. トラブルシューティング（よくあるエラー）

- **Q. `student is not in the sudoers file. This incident will be reported.` と怒られる**
  - **A.** ユーザー作成本人は自分がsudoグループにいないため、自分自身に権限を与えることができません。一度本来の管理者（rootや初期ユーザー）でログインし直し、そこから `sudo usermod -aG sudo student` を実行する必要があります。
- **Q. `sudo usermod` を打ったのに権限が変わらない！**
  - **A.** グループの変更は**「次回ログイン時」**に適用されます。コマンドを打った直後に `su - student` で入り直すか、SSHの再接続を行ってください。

## 8. 頻出メソッド・必須コマンド大全

Linuxでユーザー状態や権限を確認・操作するコマンド群です（暗記推奨）。

```bash
# 自分が誰で、どのグループにいるかを確認する（迷子になった時のお守り）
id

# ログイン中の全ユーザーを確認する
who

# ユーザーを特定のグループに安全に「追加」する（絶対に -aG を使うこと）
sudo usermod -aG <group_name> <user_name>

# rootユーザーから他のユーザーに切り替える（パスワード不要）
su - <user_name>

# /etc/passwd から自分以外の人間（一般ユーザー）を探す小技
cat /etc/passwd | grep -E "sh$|bash$"
```
