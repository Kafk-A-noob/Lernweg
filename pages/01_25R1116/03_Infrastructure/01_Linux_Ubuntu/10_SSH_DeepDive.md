# 10. SSH深掘りとOpenSSH vs PuTTY

リモートサーバーに「安全に」接続するための暗号化通信プロトコルであるSSHの本質と、2大クライアントであるOpenSSHとPuTTYの違いを徹底解説します。

## 1. SSHとは何か？

SSH (Secure Shell) は、ネットワーク越しに別のコンピューター（サーバー）を遠隔操作するためのプロトコル（通信の約束事）です。

### なぜSSHが必要なのか？

SSHが登場する以前は「Telnet」というプロトコルでリモート接続していましたが、Telnetは**通信内容（パスワード含む）が全て平文（暗号化なし）**で流れるため、同じネットワーク上にいる第三者がパケットキャプチャ（盗聴）するだけで、ログインIDとパスワードが丸見えになる極めて危険な代物でした。

SSHはこの問題を解決し、**通信の全内容を暗号化**します。たとえ通信を傍受されても、解読は事実上不可能です。

## 2. SSH認証方式の基礎

### パスワード認証（非推奨）

最もシンプルですが、最も危険な方式です。

```bash
ssh user@192.168.1.100
# → パスワードを聞かれる
```

ブルートフォース攻撃（パスワードの総当たり）の格好の的となるため、**本番サーバーでのパスワード認証は2026年現在、完全に禁止すべき**とされています。

### 公開鍵認証（必須・絶対標準）

「鍵のペア（秘密鍵 + 公開鍵）」を使う方式です。

#### 仕組みの本質（南京錠と鍵のたとえ）

1. **公開鍵** = 「南京錠（誰に渡してもよい）」→ サーバー側に置く
2. **秘密鍵** = 「その南京錠を開ける唯一の鍵（絶対に外に出さない）」→ 自分のPC内に保管

サーバーは接続要求が来ると、「この南京錠を開けられるか？」と暗号的なチャレンジを送り、秘密鍵を持っている本人だけがそれに正しく応答できます。パスワードが一切ネットワーク上を流れないため、盗聴されても安全です。

#### 鍵の生成と登録

```bash
# ed25519アルゴリズムで鍵ペアを生成する（2026年の絶対標準）
ssh-keygen -t ed25519 -C "your_email@example.com"

# 生成される2つのファイル
# ~/.ssh/id_ed25519      ← 秘密鍵（絶対に外に出さない）
# ~/.ssh/id_ed25519.pub  ← 公開鍵（サーバーに登録する）

# 公開鍵をサーバーに転送・登録する
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server_ip
```

#### パスワード認証の無効化（本番環境の鉄則）

公開鍵認証の設定が完了したら、サーバー側でパスワードログインを完全に封鎖します。

```bash
# サーバー側の設定ファイルを編集
sudo nano /etc/ssh/sshd_config

# 以下の行を変更する
PasswordAuthentication no
PermitRootLogin no          # rootユーザーでの直接ログインも禁止

# SSHサービスを再起動して設定を反映
sudo systemctl restart sshd
```

## 3. OpenSSH vs PuTTY 徹底比較

SSHで接続するための「クライアントソフト」として、2つの代表的な選択肢があります。

### OpenSSH

Linux/macOS/Windows 10以降に**標準搭載**されている、コマンドライン（CLI）ベースのSSHクライアント＆サーバーです。

```bash
# 基本的な接続
ssh user@192.168.1.100

# 秘密鍵ファイルを明示的に指定して接続（AWS EC2等）
ssh -i ~/.ssh/my-aws-key.pem ec2-user@54.xxx.xxx.xxx

# ポート番号を変更して接続（セキュリティのために22番から変更している場合）
ssh -p 2222 user@192.168.1.100

# SCPでファイルを安全に転送する（SSHの暗号化トンネルを利用）
scp ./local_file.txt user@server:/home/user/
```

### PuTTY

Windowsで長年使われてきた、**GUI（グラフィカル）ベース**のSSHクライアントです。

- **接続**: ホスト名、ポート番号、認証方式などをGUIの画面上で設定し、「Open」ボタンで接続します。
- **鍵の形式**: OpenSSHの鍵（`id_ed25519`）をそのまま使えず、PuTTY独自の `.ppk` 形式に変換する必要があります（付属の `PuTTYgen` ツールで変換）。
- **セッションの保存**: 接続先の設定を名前をつけて保存し、ワンクリックで再接続できる機能があります。

### 対比表

| 比較項目 | OpenSSH | PuTTY |
| :--- | :--- | :--- |
| **インターフェース** | CLI（コマンドライン） | GUI（グラフィカル画面） |
| **OS対応** | Linux, macOS, Windows 10+ (標準搭載) | Windows (要インストール) |
| **鍵の形式** | OpenSSH形式 (標準) | `.ppk` 形式 (独自・変換が必要) |
| **設定管理** | `~/.ssh/config` ファイルにテキストで記述 | GUIの「Saved Sessions」で保存 |
| **ファイル転送** | `scp` / `sftp` コマンド | WinSCP等の別ツールと連携 |
| **2026年の立ち位置** | **デファクトスタンダード（絶対的な主流）** | レガシー環境やGUI操作を好む層で現役 |

### どちらを使うべきか？

**結論: 2026年現在は OpenSSH 一択**です。

Windows 10以降はOpenSSHが標準搭載されたため、PuTTYをわざわざインストールする理由は「GUIで操作したい」という個人的な好み以外にはほぼありません。
特に、WSL (Windows上のLinux環境) を使う開発者は、WSL内のOpenSSHをそのまま使うのが最もシームレスです。

ただし、社内の古いWindows端末（Windows 7等）からサーバーに接続する業務が残っている現場では、PuTTYが現役で使われていることもあります。

## 4. 実務でのアンチパターンと失敗例

- ❌ **秘密鍵をSlackやメールで送信する**
  - 秘密鍵は「あなたの家の鍵そのもの」です。絶対にネットワーク上に流してはいけません。新しいメンバーには「各自で鍵ペアを生成させ、公開鍵だけを受け取ってサーバーに登録する」のが正しい手順です。
- ❌ **秘密鍵のパーミッションがガバガバ（`chmod 777`等）**
  - OpenSSHは、秘密鍵ファイルのパーミッションが `600`（所有者のみ読み書き可能）以外だと、セキュリティリスクを検知して接続を拒否します（`WARNING: UNPROTECTED PRIVATE KEY FILE!`）。`chmod 600 ~/.ssh/id_ed25519` で修正してください。

## 5. SSH Config による接続の効率化

毎回長いコマンドを打つのはつらいので、`~/.ssh/config` ファイルにエイリアス（ショートカット）を定義します。

```
# ~/.ssh/config
Host myserver
    HostName 192.168.1.100
    User deploy_user
    IdentityFile ~/.ssh/id_ed25519
    Port 22

Host aws-prod
    HostName 54.xxx.xxx.xxx
    User ec2-user
    IdentityFile ~/.ssh/my-aws-key.pem
    Port 22
```

これにより、以下のシンプルなコマンドだけで接続できるようになります。

```bash
ssh myserver
ssh aws-prod
```
