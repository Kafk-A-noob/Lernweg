# 02. WSLの管理とバックアップ

授業で触れられている `wsl` 関連のコマンド群は、Windows上でLinux環境（仮想マシン）を管理するための管理者用コマンドです。（Linuxの中ではなく、WindowsのPowerShell等で打ち込みます）

## 1. 状態の確認と停止

### `wsl -l -v` (list / verbose)

現在インストールされているLinuxディストリビューション（Ubuntuなど）の一覧と、現在の稼働状態（Running / Stopped）、WSLのバージョン（通常は WSL 2 です）を確認します。

### `wsl --shutdown` / `wsl --terminate マシン名`

WSLは、ウィンドウズを閉じても裏（バックグラウンド）で動き続けてメモリを消費します。
- `--shutdown`: すべてのWSLの仮想マシンを完全に終了（電源オフ）させ、メモリを解放します。
- `--terminate Ubuntu`: 特定のマシン（この場合は"Ubuntu"）だけを狙って強制終了します。

## 2. 実務最大の武器：スナップショット（バックアップ機能）

仮想環境（WSLやDocker、VMWareなど）が物理PCよりも優れている最大の点は、**システム全体の「セーブデータ」をファイルとして簡単に出し入れできる**点です。

授業でLAMP環境など複雑な設定をする前や、設定をミスしてLinuxが壊れてしまった時のために使います。

### エクスポート (バックアップの作成)

現在のLinuxの状態を丸ごとひとつの `.tar` ファイル（書庫ファイル）として固めて保存します。

```powershell
wsl --export Ubuntu C:\Backup\ubuntu_backup.tar
```

### アンレジスター (仮想マシンの削除・初期化)

壊れたマシンや、不要になったマシンを登録解除（＝削除）します。中のデータは完全に消えます。

```powershell
wsl --unregister Ubuntu
```

### インポート (バックアップからの復元)

エクスポートした `.tar` ファイルを指定して、全く同じ状態のLinuxを蘇らせます。

```powershell
# 書式: wsl --import <つけたい名前> <仮想ディスクを置く場所> <tarファイルのパス>
wsl --import Ubuntu-Restore C:\WSL\UbuntuRestore C:\Backup\ubuntu_backup.tar
```

※実務でも、新人エンジニア向けに「あらかじめ色々設定済みのWSL用tarファイル」を社内で配布して、それをインポートさせるだけで開発環境が完成する、という手法が取られることがあります。

## 3. 実務でのアンチパターンと失敗例

- ❌ **Cドライブ直下に巨大なWSLイメージをインポートしてしまう**
  - WSLの仮想ディスク（VHDXファイル）は使うほど無限に膨張します。デフォルトのCドライブ（システムドライブ）に置いたままだと、数ヶ月でCドライブの容量を食いつぶしてWindows自体がクラッシュする原因になります。
- ❌ **`--unregister` を「単なる停止」だと勘違いして打つ**
  - 名前からして「登録解除」程度に思えますが、実態は**完全削除**です。これまで構築したデータやソースコードが跡形もなく消え去ります。必ず打つ前に確認してください。

## 4. 2026年最新のベストプラクティス

現代の開発環境において、WSLをさらに安全・快適に運用するためのベストプラクティスです。

1. **仮想ディスク(vhd)の保存先をDドライブに変更する**
   エクスポートしたあと、十分な容量がある別ドライブ（Dドライブ等）にパスを指定してインポートし直すことで、Cドライブの枯渇を防ぎます。
2. **`wslconfig` によるリソースの上限設定**
   WSL2は放置するとPCのメモリを無尽蔵に食い尽くします（Vmmem問題）。Windowsのユーザーフォルダ直下に `.wslconfig` というファイルを作成し、「WSLが使えるメモリは最大8GBまで」のように上限を設定するのが必須の作法となっています。
3. **Docker Desktop ではなく WSL直接運用**
   有償化の波や動作の重さから、Docker Desktopを使わず、WSL（Ubuntu）の中に直接Docker Engineをインストールして運用するネイティブなスタイルが増えています。

## 5. トラブルシューティング（よくあるエラー）

- **Q. WSLの動きが異常に重い / vmmemプロセスのメモリ使用率が100%になる**
  - **A.** キャッシュが解放されていないか、上限設定がありません。まずはPowerShellで `wsl --shutdown` を実行して再起動してください。根本解決には `.wslconfig` で `memory=8GB` などの制限を書く必要があります。
- **Q. インポート時に「アクセスが拒否されました」等のエラーが出る**
  - **A.** 仮想ディスクの保存先として指定したフォルダの権限がないか、バックアップ用のtarファイルが壊れています。CドライブのProgram Files配下などは避け、ユーザー権限で書き込める場所（`C:\WSL\`など）を指定してください。

## 6. 頻出メソッド・必須コマンド大全

PowerShellからWSLを制御するためのコマンド一覧です（暗記推奨）。

```powershell
# 起動中の全ディストリビューションと状態の一覧確認
wsl -l -v

# WSLを完全にシャットダウン（メモリ解放・再起動したい時に必須）
wsl --shutdown

# デフォルトのLinuxディストリビューションを変更する
wsl --set-default Ubuntu

# 特定のWSLに「root」ユーザーとして強制ログインする（パスワードを忘れたときの最終手段）
wsl -d Ubuntu -u root

# WSL2の仮想ディスクの自動メモリ圧縮（Windows 11以降 / .wslconfigでの autoMemoryReclaim=dropcache 推奨）
```
