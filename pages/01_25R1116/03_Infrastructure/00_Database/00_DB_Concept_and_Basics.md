# 00. データベース(RDBMS)の原点と概念

## 1. 歴史と誕生背景

1970年代、IBMの研究者エドガー・F・コッド博士によって「リレーショナル・データモデル」という数学理論が発表されたのが全ての始まりです。
それまでのデータは「ファイルの中に階層状に保存する」だけであり、「Aさんが買った商品の一覧を出して」といった複雑な検索をするためには独自のプログラムを苦労して組む必要がありました。

これを、データを「Excelのような表（テーブル）」に分割し、それぞれを「ID（鍵）」で関連付けるという画期的な手法で解決したのがRDBMS（リレーショナルデータベース管理システム）です。そして、その表を操作するための言語として生まれたのが **SQL** です。

## 2. 概念と動作原理

### 行と列の厳格なスキーマ

RDBMSでは、データを保存する前に「この列（Column）には数字しか入らない」「この列には最大255文字まで入る」といった厳格なルール（スキーマ）を定義する必要があります。これにより、データが矛盾することなく安全に永続化されます。

### 正規化 (Normalization) と結合 (JOIN)

データを「重複なく」保存することが絶対命題です。「ユーザー情報」と「注文履歴」を同じ表に入れると情報が重複してしまうため、別々のテーブルに分けます。そして、データを取り出す時にSQLの `JOIN` 機能を使って、「必要な時だけ一時的に合体させて」引き出します。

## 3. 強みと弱み (Pros/Cons)

- **強み (Pros)**: ACID特性（トランザクション）による圧倒的なデータの安全性です。銀行のシステムやECサイトの決済など「絶対に一貫性が崩れてはいけない」システムにおいては、2026年現在でもRDBMS（MySQL, PostgreSQL, Oracle等）への信頼性は盤石であり、唯一無二の選択肢です。
- **弱み (Cons)**: 「データ構造（スキーマ）を後から変更するのが非常に大変」という弱点があります。また、何億人ものデータが入って巨大化した場合、複数のサーバーにまたがって分割する（水平分散・スケールアウト）のがアーキテクチャ上非常に難しく、コストがかかります。
  - ※この「分割と拡張の難しさ」「厳格すぎるルール」という弱点を解決するために生まれたのが、NoSQL（MongoDB等）などの新しいデータベースです。

## 4. 本当の基礎事項

- **主キー (Primary Key) と外部キー (Foreign Key)**
  すべての行には、他と絶対に被らない「出席番号（ID）」が必要です。これが**主キー**です。そして、「注文履歴のテーブル」の中に「買った人のID」を持たせることでユーザーと紐付けます。この他のテーブルの主キーを**外部キー**と呼びます。
- **ORM (Object-Relational Mapper) の存在**
  現代のWeb開発において、生（Row）のSQL文字列をプログラム中に書く機会は減りました。
  PrismaやSQLAlchemyといったORMというツールを使い、「PythonやTypeScriptのコードを書けば、裏で勝手にSQLに翻訳してDBに投げてくれる」アプローチが主流です。ただし、**N+1問題**などの深刻なパフォーマンス低下は生のSQLを知らないと解決できないため、SQLの基礎知識は全てのエンジニアにとって免除されない必須科目です。
