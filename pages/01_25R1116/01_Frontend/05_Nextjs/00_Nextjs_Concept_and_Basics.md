# 00. Next.jsの原点と概念

## 1. 歴史と誕生背景

2016年、Vercel社（当時のZeit社）によって開発されました。
Reactは非常に優れた技術でしたが、先述の通り「画面を描画するだけ」のライブラリであったため、SPA（Single Page Application）特有の大きな弱点を抱えていました。
それは「最初の画面が表示されるまで真っ白になる」「SEO（Googleの検索順位）に極めて弱い」という点です。

これを解決すべく、Next.jsは「サーバーサイドレンダリング（SSR）」という古くて新しい技術をReactに持ち込みました。画面の初回構築を**ユーザーのブラウザではなく、強力なサーバー側で行ってからHTMLを返す**ことで、爆速の初期表示と完璧なSEOを実現し、現在ではReactの公式ドキュメントで「まずはNext.jsを使え」と推奨される完全なデファクトスタンダードとなりました。

## 2. 概念と動作原理

### App Router と サーバーコンポーネント (RSC)

Next.js 13以降（2023年〜）、フロントエンドの歴史をひっくり返す「App Router」という新設計が導入されました。
従来のReactは「ブラウザ上で動く」のが大前提でしたが、App Routerでは**「基本すべてのコンポーネントはサーバーの中（バックエンド）で動く」**というパラダイムシフトが起きました（React Server Components）。

これにより、コンポーネントの中で直接 `await fetch()` してデータベースに接続できるなど、フロントエンドとバックエンドの境界線が完全に溶け合いました。

## 3. 強みと弱み (Pros/Cons)

- **強み (Pros)**: Reactの弱点（ルーティング、SEO、データ取得の冗長性）を全て解決済みの「全部込みフレームワーク」です。画像の自動最適化（`next/image`）や、Vercelへのボタン1発デプロイなど、実務で必要な機能が最初から完璧に揃っています。
- **弱み (Cons)**: 非常に機能が多いため、学習コストがReact単体に比べて跳ね上がります。特に「ここまではサーバーで動く」「ここからはユーザーのブラウザで動く ("use client")」という境界線を明確に意識してコードを書かないと、原因不明のエラーでビルドすら通らなくなります。

## 4. 本当の基礎事項

- **規約ベースのルーティング (File-system based routing)**
  Next.jsにおけるURL遷移（例: `/about`）は、コードで紐付けるのではなく、「フォルダ名」と「ファイル名」によって自動決定されます。`app/about/page.jsx` というファイルを作れば、それが自動的に `/about` のURLになります。フォルダ構造が「神（絶対の規約）」です。
- **Prerendering (事前描写)**
  Next.jsはビルド時（`npm run build`時）に、可能な限りすべてのページを「変化しない静的なHTML」として生成してしまいます。これにより、本番環境でのアクセスが異常に速くなりますが、ビルド時にデータベース等の準備ができていないとエラーでビルドが止まる要因にもなります。
