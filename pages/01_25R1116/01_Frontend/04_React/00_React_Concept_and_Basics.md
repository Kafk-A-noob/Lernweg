# 00. Reactの実務概念と原点

## 1. 歴史と誕生背景

2013年、Facebook（現Meta）のエンジニアであるJordan Walkeによって生み出されました。
当時のFacebookは強大なUIの問題を抱えていました。「メッセージが届いたのに、右上の通知アイコンの数字が更新されない」といったバグが頻発したのです。
原因は、「データ（状態）が変わった時に、画面（DOM）のどこを書き換えるべきか」を手作業のJavaScript（jQuery等）で管理していたため、複雑すぎて破綻したことでした。

この問題を解決するため、**「データが変わったら、画面の差分を計算して『勝手に』再描画する」**という宣言的UIのパラダイムを持ったReactが誕生しました。

## 2. 概念と動作原理

### Virtual DOM (仮想DOM)

画面（ブラウザのDOM）を直接書き換えるのは処理が非常に重いです。
Reactは、メモリ上に「ブラウザの画面のコピー（仮想DOM）」を持っています。
データが変わった時、Reactはまずメモリ上の仮想DOMを書き換え、古い仮想DOMと見比べて「本当に変わった部分（差分）だけ」をピックアップし、実際のブラウザ画面にパッチを当てます。これがReactの高いパフォーマンスの源泉です。

### コンポーネント指向と状態(State)

UIを「ボタン」「ヘッダー」「入力フォーム」といった再利用可能な**コンポーネント（部品）**に分割して開発します。
各コンポーネントは内部に `State`（状態）を持たせることができ、「Stateが変われば、そのコンポーネントだけが再描画（Re-render）される」というシンプルなルールで動きます。

## 3. 強みと弱み (Pros/Cons)

- **強み (Pros)**: 世界で最も使われているUIライブラリであり、企業での採用率・コミュニティの大きさは他を圧倒します。また「UIはデータの関数である (`UI = f(state)`)」という美しい設計思想により、テストがしやすく保守性が高いです。
- **弱み (Cons)**: React単体はただの「画面を描写するライブラリ」に過ぎず、ルーティング（画面遷移）やデータの取得機能を持っていません。そのため実世界のアプリを作るには外部ライブラリを組み合わせる必要があり、環境構築が非常に複雑になります。（この弱点を克服したのがNext.jsです）。
- また、再描画のルールを正確に理解しないと、**「文字を1文字打つたびに画面全体が無駄に再描画されてフリーズする」**といったパフォーマンス劣化（無限ループ）を容易に引き起こします。

## 4. 本当の基礎事項

- **Props は読み取り専用**: 親コンポーネントから子コンポーネントへ渡されたデータ（Props）は、子側で直接書き換えてはいけません。書き換えたい場合は、親側で状態（State）を更新する関数もセットで渡す必要があります。
- **JSXの正体**: Reactで書く `<div className="foo">` はHTMLではありません。裏側では `React.createElement("div", ...)` というJavaScriptの関数に変換される「ただのシンタックスシュガー」です。そのため、`class` ではなく `className` と書く必要があるなど、JSの予約語との衝突を避けるルールがあります。
