# AI_Powered_Tools (使用言語・技術スタック一覧)

本ディレクトリは、LLM（大規模言語モデル）を活用した自律型CLIツールの制作ログを管理する。

---

## 1. 開発プロジェクトと採用技術

### プロジェクト: Oncall CLI Agent

障害発生時の初動対応（オンコール）を支援するCLIエージェント。ユーザーの自然言語による質問を受け取り、Gemini APIを通じてLLMが自律的に外部APIを呼び出し、障害の原因分析と初動対応の提案を行う。

- **ランタイム**: Bun（TypeScriptネイティブ実行 + 単一バイナリ配布）
- **言語**: TypeScript（Strictモード）
- **LLM SDK**: @google/genai（Gemini 2.5 Flash）
- **バリデーション**: Zod（LLM出力の表記ゆれ吸収パイプライン）
- **セキュリティ**: 正規表現ベースのData Masking（IP, JWT, APIキーの自動マスク）

---

## 2. 実務に通じる技術選定の前提思考

上記で採用された技術スタックには、以下の実務的合理性がある。

- **Bun + TypeScriptの選択**: CLIツールの「起動→処理→終了」の高速サイクルにはBunのコールドスタート速度が最適であり、かつTypeScript Strictモードによる型安全性で「LLMの不定形な出力」に対するバリデーションの堅牢性を確保している。
- **Zodの選択**: Function Callingにおいて、LLMが生成する引数は表記ゆれが不可避であるため、`preprocess` パイプラインによる入力正規化はAIアプリケーション開発における必須の防衛策である。従来の手動 `if` 文チェックではなく、スキーマ駆動のバリデーションを採用することで、ツール追加時の拡張性を確保している。
- **Gemini API (Flash)の選択**: CLIツールの「即座に応答が欲しい」ユースケースにおいて、Flashモデルの低レイテンシ・低コストは最適解である。より複雑な推論が必要な場合はProモデルへの切り替えもSDK上で1行の変更で対応可能。
- **Data Masking の設計**: LLMプロバイダへの機密データ流出を構造的（アーキテクチャレベル）で防止する設計は、インフラ監視ツールのような機密性の高いドメインでは実務上必須のセキュリティ要件である。
