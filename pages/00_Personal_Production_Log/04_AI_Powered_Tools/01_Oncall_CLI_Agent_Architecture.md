# Oncall CLI Agent - アーキテクチャ評価

本ドキュメントは、Bun + TypeScript + Gemini APIを用いたオンコール初動支援CLIエージェントにおける、アーキテクチャ設計と技術的特徴の評価・分析ログである。

---

## 1. プロジェクトの特筆すべき要件と実装の難所

本プロジェクトにおける最も技術的なハードルが高い要件は、**LLMの非決定論的な出力を、システムが安全に処理できる決定論的なデータに変換すること**である。

従来のプログラムでは、関数に同じ入力を渡せば必ず同じ出力が返る（決定論的）。しかし、LLMが生成するFunction Callingの引数は、同じ意図であっても毎回微妙に異なる文字列が返ってくる。この「揺らぎ」をプログラムが処理できる形に統制するために、Zodの `preprocess` パイプラインを設計・実装した。

もう一つの難所は、**LLM自律エージェントの暴走防止**である。外部APIがダウンしている場合、エージェントは「情報が足りない」と判断して延々とAPIを叩き続ける無限ループに陥る。これを物理的に防止するMax-Loop Breaker機構の設計が必要だった。

---

## 2. 4層アーキテクチャの設計意図

本プロジェクトでは、関心の分離（Separation of Concerns）原則に基づき、システムを4つの独立した層に分割した。

```
┌─────────────────────────────────────────────────────────┐
│ UI層 (ui/)                                                │
│   ターミナルへの表示、カラー出力                             │
│   → 「何を表示するか」だけに責任を持つ                       │
├─────────────────────────────────────────────────────────┤
│ Agent層 (agent/)                                          │
│   LLMへのリクエスト構築、Function Callingループ制御          │
│   → 「LLMとどう会話するか」だけに責任を持つ                  │
├─────────────────────────────────────────────────────────┤
│ Security層 (security/)                                    │
│   Data Masking (IP/JWT/APIキーのサニタイズ)                 │
│   → 「LLMに渡すデータから何を隠すか」だけに責任を持つ         │
├─────────────────────────────────────────────────────────┤
│ Tool層 (tools/)                                           │
│   外部API通信、Zodバリデーション                            │
│   → 「外部サービスとどう通信するか」だけに責任を持つ           │
└─────────────────────────────────────────────────────────┘
```

### なぜ4層なのか

通常のWebアプリケーション（MVC: Model-View-Controller）は3層で十分だが、LLMアプリケーションでは**外部のLLMプロバイダにデータを送信する**という特性上、「送信前にデータから機密情報を除去する」専用のSecurity層が必須となる。
この層をアーキテクチャとして独立させることで、Prompt InjectionなどでLLMが意図せぬ動作をした場合でも、物理的に機密データの漏洩を防ぐ「最後の砦」として機能させている。

## 3. AIとの協業プロセス分析

本プロジェクトでは、以下の点でAIアシスタントとの協業が効果的に行われている。

### アーキテクチャの壁打ち

「LLMエージェントが暴走しない設計にするにはどうすればいいか」というアーキテクチャレベルの問いをAIに投げ、Max-Loop Breaker・Graceful Error Handling・Interceptionパターンといった設計パターンの選定に活用している。

### セキュリティ設計のレビュー

Data Maskingの正規表現パターン（IPv4/IPv6/JWT/メールアドレス/APIキー）の網羅性について、AIをレビュアーとして活用し、パターンの漏れや過検知のリスクを事前に洗い出している。

### Zodスキーマ設計

LLMの表記ゆれパターン（日本語のひらがな/カタカナ/英語混在）をAIとブレインストームし、`preprocess` パイプラインの正規化マッピングを設計している。

---

## 【AIメンターからの講評：実務への接続】

本プロジェクトで実装された「LLMの出力を信頼しない」という前提に立った防衛的設計は、AIプロダクト開発の現場で理論的に最も重要な原則です。

多くの開発者がLLMを「賢いAPI」として無邪気に信頼してしまう中、**「LLMは嘘をつく・暴走する・機密を漏らす」という三重のリスクに対して、それぞれ独立した防衛層を設けた**アーキテクチャは、実務のセキュリティ要件を満たすレベルに達していると評価できます。

特に、Security層をMVCの外に独立させた判断は秀逸です。実務では「セキュリティはController層の中でif文チェックすればいいでしょ」という安易な設計が横行し、結果としてセキュリティチェックの漏れやバイパスが頻発します。セキュリティを**アーキテクチャとして強制する**（バイパス不可能にする）設計は、インフラ・SREの現場でも通用する考え方です。
