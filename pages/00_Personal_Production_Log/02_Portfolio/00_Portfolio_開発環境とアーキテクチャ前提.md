# Portfolio (開発環境とアーキテクチャ前提)

本ディレクトリは、就職活動における中核となるWeb3Dポートフォリオ「Project Last-Stand」の技術スタック、およびそのアーキテクチャ選定の前提思想を管理する。

---

## 1. 採用技術スタックと選定理由

本プロジェクトは、視覚的なインパクト (3D表現) と、Webアプリケーションとしての堅牢性・保守性の両立を目指し、以下のモダンスタックを採用している。

- **コアフレームワーク**: Next.js 15+ (App Router)
  - 選定理由: ルーティング管理の自動化、サーバーコンポーネントによる初期ロードの高速化、およびVercelとのシームレスなデプロイ連携。
- **3Dグラフィックス**: Three.js / React Three Fiber (R3F)
  - 選定理由: 従来の生WebGLや生のThree.jsを用いた命令的記述から脱却し、Reactの「宣言的UIコンポーネント」として3Dオブジェクトを扱うため。これにより状態管理 (React Hooks) との統合が極めて容易になる。
- **グローバル状態管理**: Zustand
  - 選定理由: Reduxのような巨大なボイラープレート (定型文) を避けつつ、React Contextよりも軽量かつ再レンダリングを抑えたグローバル状態管理を実現するため。3Dのロード状態やUIの切り替えフラグ管理に使用。

---

## 2. アーキテクチャの特徴的な実装

### ① データ駆動型UI (Data-Driven UI)

「3Dモデルを追加するたびにReactのコードを書き換える」という運用は保守性が低い。
これを防ぐため、`asset-manifest.ts` という単一の設定ファイルを用意し、そこにモデルのパスやメタデータを追記するだけで、UI側のナビゲーションや3Dキャンバス上のローダーが自動的にループ処理 (`.map()`等) で生成される設計となっている (関心の分離)。

### ② Optimistic UI (楽観的UI) と SmartLoader

3Dアセット (.glb等) はどうしてもファイルサイズが大きくなり、ロード待ちが発生する。
ユーザーに「遅い・動かない」とストレスを感じさせないため、以下の工夫を行っている。

1. **Optimistic UI**: 3Dの読み込みが終わるのを持たず、クリックした瞬間にHTMLレイヤーの文字やUIだけは即座に切り替わるようにし、体感速度を上げる。
2. **SmartLoader**: Vercel環境特有の「ストリーミング中のファイル総量(Total size)取得不能問題」に対し、アセットマニフェストにあらかじめファイルサイズをメタデータとして持たせておくことで、正確なロード進捗 (%) を計算・表示させる回避策を実装している。

### ③ 3Dアセットの最適化 (Draco Compression)

Webブラウザ上で3Dを動かす際の最大のボトルネックである「通信容量」を劇的に改善するため、Googleが策定したDraco圧縮規格を用いてアセットを極限まで軽量化している。ローダー側にもDraco対応のデコーダーを組み込むことで、高品質なモデル表示とパフォーマンの維持を両立している。
