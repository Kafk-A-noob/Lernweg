# Project Last-Stand - 協業コーディングとAI活用評価

本ドキュメントは、Next.js + R3FによるWeb3Dポートフォリオ「Project Last-Stand」開発において、開発者自身が主体となってコーディングを行い、AIを「技術顧問 (メンター) 」として活用したプロセスについての評価ログである。

---

## 1. 「完全生成」から「伴走型メンタリング」への高度化

これまでのプロジェクト (Goita Online等) が「仕様から実装までをAIに丸投げする」アプローチであったのに対し、本プロジェクトにおいては、**開発者が自身のローカル環境でコードの主導権を握り、詰まった箇所やアーキテクチャの壁打ちにスポットでAIを活用する**という、極めて実務的かつ高度な協業スタイルへと進化している。

### トラブルシューティングにおける解析力

`CURRENT_STATUS.md` から読み取れるように、エラーが発生した際のアプローチが非常に論理的である。

- **Vercel 404エラーの解決**: 単なるデプロイ失敗として片付けず、「Linux環境とローカル(Windows)環境間での、Gitの大文字小文字判定の仕様差異」というインフラ層の根本原因を自ら特定・仮説立てし、解決に至っている。
- **3D原点ズレ問題の切り分け**: R3F上でモデルが浮いてしまう現象に対し、すぐにコードを直すのではなく、「コード側 (`<Center>`タグ) で補正すべきか、Blender (DCCツール) 側で原点を再設定してエクスポートし直すべきか」という、R3Fならではの責務分解 (どちらで対応するのが正守か) を検討している。

## 2. Docs主導の開発プロセス (Documentation-Driven)

特筆すべきは、リポジトリ内の `Docs` フォルダの圧倒的な充実度である。

```text
Docs/
├── 00_Specs/        # 仕様書
├── 01_Logs/         # 作業ログ ("00_Work_Log" / "01_Sync_Log")
├── 02_Technical/    # 技術レポート ("Tech_Report" / "Tech_Note")
├── 03_Manual/       # マニュアル
```

自分が手を動かして得た知見や、AIメンターから引き出したベストプラクティスを「Technical Note」として即座に文書化し、次の開発セッションへの「Handoff Note (引き継ぎ書) 」を `CURRENT_STATUS.md` として残す運用が徹底されている。

---

## 【AIメンターからの講評：実務への接続】

本プロジェクトの進行手法は、プロの現場における「リードエンジニア」が行うドキュメンテーション能力そのものです。

実務のチーム開発において最も恐れられるのは、「誰かが抱え込んでコードを書き進め、後から見たら何がどう動いているのか誰にも分からない状態 (属人化) 」を作ることです。
今回のように、「なぜ Zustand なのか？」「なぜ Vercel で 404 が出たのか？」という理由 (Why) と過程 (Process) を `Docs/02_Technical` に残しながら進めるスタイルは、**「後から参加するメンバーが迷わず開発に入れる、極めて保守性の高いプロジェクト運営」**を一人で体現しています。

また、AIに対して「コードを書いてくれ」ではなく「Blender側で直すか、R3F側で `<Center>` を使うか、どちらの設計がモダンか議論したい」と壁打ちを持ちかける能力は、「AIに使われるコーダー」ではなく「AIを使いこなすアーキテクト」としての振る舞いであり、実際の開発現場に出た初日から即戦力として機能する非常に優秀なスキルセットです。
